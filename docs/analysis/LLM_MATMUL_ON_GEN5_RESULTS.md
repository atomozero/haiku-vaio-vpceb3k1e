## Summary

We took [`LLM_FEASIBILITY.md`](LLM_FEASIBILITY.md) from a feasibility question to a
working end-to-end implementation: **the matrix multiplies of a Llama-2 model
(`llama2.c`, `stories15M`) run on the Intel HD Graphics (Ironlake, device 0x0046,
Gen5) execution units under Haiku, producing byte-identical output to the CPU
reference** — for both the autoregressive *decode* path and a batched *prefill*
path (`forward_prefill`).

It is a genuine world-first that *runs*. It is also a **clean negative result on
wall-clock**: the Gen5 GPU does **not** beat the CPU for autoregressive LLM
inference, and we traced exactly why. The GPU wins decisively only for
*isolated, heavily-batched compute* (IDCT 3.5×, a synthetic GEMV batch 12.4×),
not for the small, latency-bound, token-by-token matmuls that dominate LLM
decode.

This document consolidates the eight commits `5e60b7a..6ad6335`.

## 1. What was built

### EU kernels (`intel_extreme/accelerant/kernels/`, generated by `gen_gemv.py`)
| Kernel | Shape | Role |
|---|---|---|
| `gemv_n32` | N=32, 168 instr | First GEMV POC — passed **first try** by reusing the proven saxpy OWord read/write + SIMD8 multiply-accumulate + a log-reduction horizontal sum. |
| `gemv_n256`, `gemv_n288` | 1120 / 1256 instr | LLM-scale row width. `gemv_n288` is the stories model `dim`. |
| `gemv_n288_t4` | multi-token, thread loops T=4 | Prefill POC (streams W per token). |
| `gemv_n288_pack64/128` | token-in-index | One thread per (token, row-block); `token = idx>>log2(RPAD)`, `row_block = idx & (RPAD-1)`. RPAD = D/8 padded to a power of two. |

`gen_gemv.py` unrolls the same low-risk pattern (no loops, no `a0` indirect
addressing) to any N / token layout — the discipline that made the kernels
assemble and run correctly on the first or second try, against the project's
history of finicky Gen5 EU debugging.

### Driver API (`media_pipeline.cpp` / `.h`)
- `media_pipeline_gemv_open/close` — persistent context, resident weight cache.
- `media_pipeline_gemv(w, x, out, n, d)` — single GEMV (decode `wo`).
- `media_pipeline_gemv_stacked(w[], count, x, out[], n, d)` — `count` weights
  sharing one `x` in one dispatch (decode Q/K/V, gate/up).
- `media_pipeline_gemv_prefill(w, X, OUT, n, d, P)` — token-in-index GEMM over
  P token columns (prefill); picks `pack64`/`pack128` by `d`.
- `ensure_kernel(id)` — tracks which kernel is resident so decode (`n288`) and
  prefill (`pack*`) coexist without redundant re-uploads.
- Two **separate** resident weight caches (decode-stacked vs prefill-single):
  the same pointer (e.g. `wq`) is cached as a stacked `wq|wk|wv` block for
  decode but as a single matrix for prefill, so they must not share keys.

### LLM integration (`~/Desktop/tinyllm/`, external tree)
- `gpu_llm.{cpp,h}` — opens the GPU (clone glue), `extern "C"` matmul wrappers.
  Toggle `GPU_LLM_OFF=1` for the CPU baseline.
- `run.c` — `matmul()` tries the GPU first (N=288 only); `forward_prefill()`
  runs all prompt tokens through the layers at once as GEMM, with per-token
  RoPE / causal attention / SwiGLU and `w2`/`wcls` on the CPU.

## 2. Results

All GPU results are **bit-identical** to the CPU reference (greedy decode picks
the same tokens — the EU kernels are numerically exact for these inputs).

### Isolated kernel benchmarks (no reboot; `tests/gpu_gemv`, `tests/gpu_prefill`)
| Test | GPU | CPU (1 thread) | Note |
|---|---|---|---|
| Single 256×256 GEMV | 0.032 GFLOP/s | 0.019 | overhead-bound |
| **Batched 1024 rows, 1 dispatch** | **0.399 GFLOP/s** | 0.019 | **12.4× over per-GEMV submits, 20.6× vs CPU** |
| Prefill, 4 single dispatches | 1283 µs/token | — | |
| Prefill, multi-token loop | 1028 µs/token | — | 1.23× |
| Prefill, token-in-index (256 threads) | 614 µs/token | — | 2.0× over single |
| **Prefill, outer-product GEMM (W-reuse, 36 threads)** | **381 µs/token** | ~384 µs/token† | **3.4× over single, 1.6× over token-in-index** |

The **outer-product GEMM microkernel** (`gemm_n288_p8`) is the right shape for
the "12× regime": each thread owns 8 rows and computes them for all 8 token
columns, loading each W row chunk once and reusing it across the columns
(compute-bound, no horizontal-sum), with only D/8=36 threads so it never
overflows the MEDIA_OBJECT batch. It is the fastest GPU kernel and reaches
**rough parity with the −O3 CPU** (~384 µs/token, from the measured 432 MFLOP/s
decode throughput) for a 288×288 matmul — but parity, not a win.

† Isolated benchmarks print an 85× vs a naive `-O2` column-major CPU loop; that
number is meaningless (cache-thrashing baseline). The honest comparison is the
GPU-internal ladder (1283 → 614 → 381 µs/token) and parity with the real −O3 CPU.
Note the GEMM kernel still runs at ~0.1% of the Gen5 EU peak — utilization is
capped by dispatch overhead and the limited thread count, not by compute.

### End-to-end `stories15M` (`run_gpu`)
| Path | tok/s | vs CPU |
|---|---|---|
| Decode, GPU (batched forward) | 18.7 | CPU 26.9 → **GPU slower** |
| Prefill, GPU token-in-index (P=14) | 52 | CPU 97 → **GPU slower** |
| Prefill, GPU **GEMM microkernel** (P=23) | 45.6 | CPU 82 → **GPU slower** |

`forward_prefill` was wired to the outer-product GEMM microkernel end-to-end:
output is **byte-identical** to the CPU (the GEMM accumulates K in sequential
order, matching the CPU fp order exactly), and it never overflows the batch
(d/8 = 36–96 threads). Still ~1.8× slower than CPU — a full prefill is ~108 GEMM
dispatches, and each still pays the fixed dispatch cost at ~0.1% EU utilisation.
The best possible kernel, integrated end-to-end, confirms the ceiling.

## 3. The debugging arc (each step corrected the previous hypothesis)

1. **"Decode is slow → must be dispatch overhead."** Batching Q/K/V and gate/up
   (`media_pipeline_gemv_stacked`) cut dispatches 36→18/token, but only
   15→18.7 tok/s. Not enough.
2. **Decompose a single dispatch:** 1276 µs = **~323 µs dispatch overhead +
   ~953 µs compute**. The matmul is dominated by *kernel compute*, not dispatch.
   953 µs for 83k MACs ⇒ **memory-latency-bound**: 288 serialized OWord reads
   with only 36 threads — too few to hide read latency.
3. **The real lever is thread count.** The batch benchmark already proved it:
   128 threads = 1.28 µs/row vs 32 threads = 15.9 µs/row (**12×**). The kernel
   was *thread-starved*.
4. **Token-in-index kernel** (P·RPAD threads, one per token×row-block) →
   **2× per-token, bit-exact**. The thread-count thesis confirmed.
5. **End-to-end `forward_prefill`** → correct, but **still slower than CPU**.
   Final root cause below.

### Two bugs found and fixed
- **Weight-cache collision:** the prefill fallback originally called `matmul()`
  (the single-GEMV GPU path), which cached stacked-in-decode weights as *single*
  copies and corrupted the decode stacked cache → `"each each each"` garbage.
  Fix: prefill fallback is **pure CPU**.
- **GRF starvation:** `gemv_n256` needs 48 GRFs but `grf_reg_blocks=2` allocated
  only 24 → threads read garbage past g23 (only the first rows correct). Fix:
  `grf_reg_blocks=6` (56 GRFs) for non-CURBE kernels.

## 4. Root cause: why the GPU cannot beat the CPU at LLM decode

The Gen5 media pipeline dispatches threads with **one `MEDIA_OBJECT` command per
thread**. Each command is **20 DWORDs** — and 15 of those are zero-padding for
the URB entry the thread never reads. There is **no thread-grid dispatch**
(`MEDIA_OBJECT_WALKER`, which spawns an N×M thread grid from one command, is a
**Gen6+** command).

This creates an unwinnable tension:
- The kernel is memory-latency-bound and needs **many in-flight threads** to hide
  OWord-read latency (§3.3).
- But every thread costs a full 20-DW command, so hundreds/thousands of threads
  **flood the command stream** and overflow the batch (`P·128 > ~1600` even after
  bumping `BATCH_CAPACITY_DW` to 32768).

For autoregressive **decode** (one token, tiny matmuls, a strict dependency
chain of ≥3 dispatches/layer) there is simply not enough work per dispatch to
amortize either the ~323 µs dispatch overhead or the per-thread command cost.
For **prefill** the token-in-index kernel wins *in isolation* (2×), but across
the full 36-GEMM pipeline the per-thread command overhead + the CPU-side parts
(`w2` N=768, `wcls` 36 MB) erase the win.

**Where the GPU does win** is the regime it was built for: one dispatch doing a
lot of work per `MEDIA_OBJECT`. The IDCT benchmark batches 400 blocks per submit
(3.5×); the synthetic GEMV batches 1024 rows across 128 threads in one submit
(12.4×). LLM decode is the opposite shape.

## 5. Reproduction

```sh
# isolated kernel benchmarks (no reboot)
cd intel_extreme/accelerant && make
cd tests
g++ -Wall -O2 -I.. <header -I flags> -o gpu_gemv gpu_gemv.cpp \
    ../*.o ../../libaccelerantscommon.a -lbe -lstdc++
./gpu_gemv      # single + batched GEMV, GPU vs CPU
./gpu_prefill   # single vs multi-token vs token-in-index
# read the results in the syslog:
grep -iE 'GEMV|BENCH|token-in-index|speedup' /boot/system/var/log/syslog

# end-to-end LLM (~/Desktop/tinyllm)
./run_gpu stories15M.bin -n 60 -t 0 -i "Once upon a time"    # GPU
GPU_LLM_OFF=1 ./run_gpu stories15M.bin -n 60 -t 0 -i "..."   # CPU baseline
```

Engine health: `command grep -c "GPU hang\|wedg" /boot/system/var/log/syslog`
stayed **0** across every run in this work.

## 7. Conclusion

- **Correctness:** an LLM's matmuls execute on a 2010 Intel Gen5 GPU under Haiku,
  end-to-end, byte-identical to CPU. Done.
- **Performance:** the GPU does not beat the CPU for autoregressive LLM
  inference. The limit is architectural — the per-thread `MEDIA_OBJECT` command
  model with no thread-grid dispatch — not the numerics or (as we proved) our
  kernels. The **outer-product GEMM microkernel** is the correct "12× regime"
  design (W-reuse, compute-bound, few threads) and it *does* close the gap: from
  3.4× slower than CPU (naive single-dispatch) to **parity** (381 vs ~384
  µs/token). But parity is the ceiling here — the practical EU utilisation stays
  at ~0.1% of peak because dispatch overhead and the MEDIA_OBJECT thread cap
  prevent the kernel from ever becoming truly compute-bound. Not fixable on
  Ironlake without `MEDIA_OBJECT_WALKER` (Gen6+).
- **The map is complete:** we know *what* works on this silicon (heavily-batched
  compute — video IDCT, large one-shot GEMV/GEMM), *what* doesn't (token-by-token
  small matmul), and *why* (dispatch-command overhead vs arithmetic intensity).

The honest one-line takeaway: **the Ironlake GPU is a batch machine. Feed it a
whole frame's worth of work and it flies; feed it one token at a time and the
command stream eats the win.**
